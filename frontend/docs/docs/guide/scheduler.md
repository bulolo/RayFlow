# 任务调度

任务调度用于编排 **开发运维内的 Flink 作业**。它的核心不是替代 Flink，而是把多个已经登记好的 SQL/JAR 作业串成一个可审计的 DAG 工作流。

![任务调度](/screenshots/5.png)

## 适用场景

- ODS -> DWD -> DWS -> ADS 批处理链路。
- Paimon 表构建后的下游 StarRocks 刷新任务。
- 多个 Flink 批任务必须按顺序执行的场景。
- 需要保留每次执行记录、节点状态和运行日志的场景。

## 核心概念

| 概念 | 说明 |
| --- | --- |
| 工作流 | 一组可运行的 DAG 编排定义。 |
| 节点 | 一个开发运维里的 Flink SQL/JAR 作业。 |
| 边 | 节点间依赖关系，上游成功后才会启动下游。 |
| 执行记录 | 每次手动运行生成一条 workflow execution。 |
| 节点执行 | 每个节点单独记录开始、结束、状态和错误。 |
| 执行日志 | 后端调度循环写入的审计日志，用于定位运行过程。 |

## 编排方式

1. 在开发运维中先创建并验证 Flink 作业。
2. 进入任务调度，创建工作流。
3. 从左侧作业池拖拽作业到画布。
4. 连接节点形成 DAG。
5. 保存 DAG 编排。
6. 点击运行工作流。
7. 在运行日志中查看执行状态和节点日志。

## 执行语义

当前调度执行是异步串行推进：

- 没有前置依赖的节点可以进入待运行。
- 节点提交后，调度循环会轮询该 Flink 作业状态。
- 当前节点成功完成后，下游节点才会继续。
- 如果节点失败，工作流会按节点重试配置处理；超过重试次数后工作流失败。

## 如何判断 Flink 完成

RayFlow 通过 Flink REST API 查询作业状态：

- 批任务进入 `FINISHED` 视为成功完成。
- 流任务通常不会自然结束，不适合作为串行 DAG 的中间节点。
- `FAILED`、`CANCELED` 等终态会进入失败或取消处理。

因此当前 Demo 更适合批任务链路。

## Demo 工作流

`make init-test` 会初始化 `scheduler-demo-user-behavior-pipeline`：

```text
ods-user-log-sync -> dwd-click-event-cleanup -> dws-user-behavior-hourly
```

三个节点都是批处理示例，作业位于 `demo/batch/scheduler` 分组。整体执行时间做了差异化，便于观察异步运行、节点推进和取消。

## 当前边界

- 定时任务暂不作为当前闭环重点，先保证手动运行、状态推进、日志和 UI 反馈完整。
- 流任务可被加入节点，但如果它长期运行，后续节点不会自然推进。
- 工作流编排面向平台内 Flink 作业，不直接编排外部 shell、HTTP 或数据库任务。
